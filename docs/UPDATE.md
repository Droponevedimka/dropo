# dropo — обновления и зависимости

Как устроены релизы, обновление приложения и загрузка зависимостей. Источник
истины для версий — **GitHub-релизы репозитория
[Droponevedimka/dropo](https://github.com/Droponevedimka/dropo)**.

## Проблема, которую решаем

Тяжёлые бинарные зависимости (`bin/`: sing-box, winws+WinDivert, WireGuard, xray,
tg-ws-proxy, фильтры, payload-файлы) — это ~68 МБ из ~68 МБ релиза. При этом они
меняются **редко** (только когда бампается версия инструмента), а само приложение
(`dropo.exe` + UI) меняется **часто**. Тащить 68 МБ при каждом обновлении и хранить
их в git — расточительно. Поэтому **разделяем** приложение и зависимости.

## Два независимых артефакта релиза

Каждый GitHub-релиз содержит:

| Артефакт | Что внутри | Размер | Меняется |
|---|---|---|---|
| `dropo-<version>-<hash>.zip` | `dropo.exe`, `resources/`, `licenses/`, скрипты, `dependencies.json` (манифест). **Без `bin/`** | ~12 МБ | часто (логика/UI) |
| `dependencies-<depsVersion>.zip` | весь `bin/` (тяжёлые зависимости) + маркер `bin/.deps-version` | ~68 МБ | редко (бамп инструмента) |

- **`version`** — версия приложения из `version.json`, бампается на каждое изменение.
- **`depsVersion`** — детерминированный короткий хэш набора версий инструментов
  (sing-box|wireguard|wintun|byedpi|spoofdpi|zapret|tgwsproxy|xray). Меняется
  только когда реально меняется какой-то бинарь. Несколько релизов приложения
  могут переиспользовать **один и тот же** `dependencies-<depsVersion>.zip`
  (заливается один раз).

`dependencies.json` (лежит рядом с `dropo.exe` в app-архиве):
```json
{
  "depsVersion": "a1b2c3d4e5f6",
  "asset": "dependencies-a1b2c3d4e5f6.zip",
  "sha256": "…",
  "size": 71600000,
  "appVersion": "2.0.2",
  "repo": "Droponevedimka/dropo"
}
```

## Зависимости НЕ в git

`dependencies/` в `.gitignore`. `build.ps1` скачивает их в этот локальный кэш на
этапе сборки (как и раньше), но в репозиторий они не попадают, а пользователю
едут отдельным релиз-ассетом. Источник для пользователя — **наш релиз**, а не
апстримы (sing-box/zapret/…): контролируемо, воспроизводимо, без зависимости от
доступности и rate-limit сторонних репозиториев.

## Сценарий: первый запуск (portable, без `bin/`)

1. Пользователь распаковал `dropo-<version>-<hash>.zip` (≈12 МБ, без `bin/`).
2. На старте `EnsureDependencies()`:
   - читает требуемый `depsVersion` из `dependencies.json`;
   - читает установленный из `bin/.deps-version` (маркер);
   - если `bin/` отсутствует/неполон **или** маркер ≠ требуемому →
     - находит ассет `dependencies-<depsVersion>.zip` в релизах репозитория
       (или берёт прямую ссылку из манифеста), скачивает во временную папку,
       **проверяет sha256**, распаковывает в `bin/`, пишет маркер;
     - показывает прогресс («Загрузка компонентов…»), блокирует запуск VPN до готовности.
   - если маркер совпал → **ничего не качаем** (это и есть «не качать постоянно»).
3. Дальше приложение работает как обычно.

## Сценарий: обновление приложения

1. `CheckForUpdates()` опрашивает `releases/latest` репозитория, сравнивает
   `version` (semver).
2. Если новее → скачивает `dropo-<newVersion>-<hash>.zip`, распаковывает,
   подменяет `dropo.exe` + `resources/` (self-update через хелпер, который
   завершает старый процесс и переносит файлы), `bin/` **не трогает**.
3. После рестарта новый `dependencies.json` может указывать новый `depsVersion`:
   - совпал с установленным → деплой мгновенный, ничего не качаем;
   - изменился → `EnsureDependencies()` один раз докачивает новый
     `dependencies-<depsVersion>.zip`.

Итог: апдейты приложения маленькие и частые; зависимости докачиваются только
когда реально поменялись.

## Что делает сборка (`build.ps1`)

1. Собирает `dropo.exe`, копирует `bin/`, `resources/`, `licenses/`, скрипты — как раньше (полная папка релиза для локального запуска/тестов остаётся со всем `bin/`).
2. Считает `depsVersion` = SHA256(набор версий инструментов).Substring(12).
3. Пишет маркер `bin/.deps-version` = `depsVersion`.
4. Создаёт `dependencies-<depsVersion>.zip` из `bin/` (+ маркер).
5. Пишет `dependencies.json` (manifest со sha256/size ассета) рядом с `dropo.exe`.
6. Создаёт app-архив `dropo-<version>-<hash>.zip` из папки релиза **без `bin/`**.
7. Оба архива публикуются в GitHub-релиз (app — на каждый релиз; deps — только
   если `depsVersion` ещё не залит).

## Граничные случаи и устойчивость

- **Нет сети на первом запуске** → приложение показывает понятную ошибку и
  предлагает повторить; без `bin/` VPN не стартует (нечем).
- **Битый/неполный `bin/`** → проверка ключевых бинарей + sha256 архива;
  при несовпадении — перекачка.
- **Откат версии** → `depsVersion` совпадёт со старым → перекачки нет.
- **Полностью офлайн-дистрибутив** (если нужен) → можно отдать оба архива вместе
  или собрать «fat»-zip; по умолчанию — раздельно.
- **GitHub API rate-limit** → ассет ищется по релизам; URL можно зашить прямой
  (`releases/download/<tag>/<asset>`), чтобы не зависеть от API.

## Две сборки: тестовая и продовая

| | `build.ps1` (тестовая) | `release.ps1` (продовая) |
|---|---|---|
| Назначение | локальные тесты | публикация релиза |
| В git | да (трекается) | **нет** (в `.gitignore`, только у владельца) |
| Версия | `version.json` | `version.json` |
| Имя app-архива | `dropo-<version>-<hash>.zip` (с хэшем) | `dropo-Windows-Portable-x64.zip` (без хэша) |
| GitHub-релиз | не создаёт | создаёт тег `v<version>` + заливает ассеты |
| Если версия уже в релизах | — | **останавливается**: «обновите version.json» |

Версия бампается **вручную** в `version.json` (по требованию — лучше руками).
`release.ps1` ориентируется на неё и не перезаписывает существующий релиз.

## Имена ассетов (Hiddify-style, платформа в имени)

Платформа всегда в имени — чтобы один релиз держал ассеты всех ОС:

| Платформа | App-ассет | Deps-ассет | Статус |
|---|---|---|---|
| **Windows** (portable only) | `dropo-Windows-Portable-x64.zip` | `dropo-Windows-Dependencies-x64-<depsVersion>.zip` | реализовано |
| Android | `dropo-Android-universal.apk` | — (бандл) | позже |
| macOS | `dropo-MacOS-arm64.dmg` | — | позже |
| Linux/Ubuntu | `dropo-Linux-x64.AppImage` / `.deb` | — | позже |
| iOS | `dropo-iOS.ipa` | — | позже |

`dependencies.json` в app-архиве каждой платформы ссылается на свой deps-ассет;
bootstrap качает `releases/download/v<version>/<asset>` (fallback — поиск по API).

## Продовый релиз — GitHub Actions (по бампу version.json)

Ты **меняешь только `version` в `version.json`** и пушишь в `main`. Воркфлоу
[.github/workflows/release.yml] сам решает, релизить или нет. Локальный `gh` не нужен,
теги вручную ставить не нужно.

```powershell
# единственное ручное действие:
#   version.json -> "version": "2.0.4"
git add version.json; git commit -m "v2.0.4"; git push origin main
```

Воркфлоу (триггер — изменение `version.json` в `main`, плюс ручной `workflow_dispatch`):
1. читает `version.json` → `v<version>`;
2. **проверяет, есть ли уже релиз с этим тегом** — если есть, ничего не делает
   (нужно поднять версию);
3. иначе ставит Go/Flutter, делает `build.ps1 -AppOnly` (только приложение,
   **без** `dependencies/` — clean-runner), создаёт тег `v<version>` из текущего
   `main` и публикует релиз с `dropo-Windows-Portable-x64.zip`.

### Тяжёлый deps-архив (редко меняется)

`build.ps1 -AppOnly` берёт идентичность deps-архива из **`deps-lock.json`**
(коммитится) и пишет её в манифест приложения. Сам `dropo-Windows-Dependencies-x64-<depsVersion>.zip`
**не** собирается в CI (его негде взять на чистом раннере). Воркфлоу **переиспользует**
уже залитый архив: находит его по имени в прошлых релизах и **прикладывает** к новому,
чтобы страница релиза была самодостаточной (как у Hiddify). Если архив ещё нигде
не залит — релиз выходит только с приложением + предупреждение (клиент всё равно
найдёт deps по имени по всем релизам).

Когда зависимости меняются (бамп инструмента в `version.json`):
1. локально `./build.ps1` (полная) → новый `dropo-Windows-Dependencies-x64-<новый>.zip`
   и обновлённый `deps-lock.json`;
2. залейте этот deps-архив в любой релиз **один раз** и закоммитьте `deps-lock.json`;
3. дальше CI сам прикладывает его к новым релизам.

> **Первый релиз** (`depsVersion` ещё не залит): после публикации залейте deps-архив
> текущего `depsVersion` в релиз один раз (`gh release upload v<version> <путь к zip>`
> или вручную через web). Дальше CI переиспользует его автоматически.

### Локальный fallback (`release.ps1`, owner-only, в .gitignore)

Если нужно опубликовать без Actions: `./release.ps1` (требует `gh auth login`) —
собирает и заливает app + deps. Скрипт не в репозитории (только у владельца).
